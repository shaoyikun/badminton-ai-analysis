# 羽毛球 AI 动作分析项目技术方案 V1

## 1. 文档定位
本技术方案以 `docs/prd/PRD.md` 和 `docs/design/INTERACTION-DESIGN.md` 为输入，目标是支撑 MVP 阶段的产品实现。

---

## 2. 技术目标
围绕交互设计，技术层需要完成以下能力：
1. 支持视频上传与预览
2. 支持异步分析任务
3. 支持视频质量校验
4. 支持正手高远球的姿态识别与问题诊断
5. 支持基于真实姿态证据的结构化报告返回
6. 支持历史记录与复测数据存储

---

## 3. 推荐总体架构
移动端前端应用
→ API 服务
→ 文件存储
→ 视频预处理服务
→ 姿态识别服务
→ 动作评估服务
→ 报告生成服务
→ 结果存储
→ 返回前端

### 3.1 平台策略
首发阶段默认按移动端实现，重点考虑：
- 手机视频上传链路
- 移动网络环境下的上传稳定性
- 竖屏页面状态切换
- 移动端报告展示的信息密度控制

---

## 4. 模块设计

### 4.1 前端层
对应交互文档中的：
- 首页
- 拍摄指引页
- 上传页
- 分析中页
- 报告页
- 历史记录页
- 复测对比页

技术职责：
- 上传视频
- 轮询分析状态
- 展示报告
- 展示异常提示

---

### 4.2 API 层
建议提供以下接口：
- 创建分析任务
- 上传视频
- 查询任务状态
- 获取分析结果
- 获取历史记录
- 获取复测对比结果

---

### 4.3 视频预处理层
职责：
- 视频格式统一
- 抽帧
- 时长校验
- 画面质量检测
- 主体完整度与姿态覆盖率前置判断

输出：
- 是否可进入分析
- 错误原因分类
- 关键帧序列

---

### 4.4 姿态识别层
职责：
- 提取人体关键点
- 形成时间序列骨架数据
- 输出可解释的质量摘要与稳定帧统计

MVP 推荐策略：
- 先用成熟 2D 姿态估计方案起步
- 正式分析当前支持正手高远球和杀球

---

### 4.5 动作评估层
职责：
- 根据关键点和时序特征计算维度评分
- 识别当前最关键的问题与次级问题
- 生成结构化问题标签与影响说明

推荐方式：
- 规则引擎 + 明确质量门槛
- 暂不建议直接依赖纯大模型端到端判断

---

### 4.6 报告生成层
职责：
- 将结构化结果映射成报告字段
- 生成问题解释
- 生成标准动作差异说明
- 生成复测反馈与下次关注点

推荐方式：
- 问题标签体系 + 复测反馈模板库
- LLM 负责润色，而不是直接无约束生成

---

### 4.7 数据存储层
需保存：
- 原始视频地址
- 上传元数据
- 任务状态
- 分析结果
- 历史记录
- 复测关系

---

## 5. 数据结构建议

### 分析结果建议字段
- action_type
- total_score
- dimension_scores
- issues
- suggestions（短期语义收口为复测关注点 / 后续观察建议）
- comparison
- standard_comparison
- retest_advice
- keyframe_refs
- scoring_evidence
- created_at

### 错误结果建议字段
- error_code
- error_message
- retry_suggestion
- error_category

---

## 6. 技术选型建议

### 前端
MVP 可先采用轻量 Web 方案，后续再迁移到更正式的前端框架。

### 后端
建议采用便于快速迭代的 Web API 框架。

### 姿态识别
优先选择成熟、现成、可快速验证的方案。

### 存储
视频文件与分析结果分离存储。

---

## 7. 分阶段实施建议

### Phase 1：产品演示版
- 前端静态 demo
- 模拟报告输出
- 跑通交互流程

### Phase 2：可用 MVP
- 接真实上传
- 接视频预处理
- 接姿态识别
- 接规则评分
- 接结构化报告
- 证据不足时明确拒绝，不再回退到 mock 报告

### Phase 3：增强版本
- 继续优化历史记录与复测对比表达
- 增加动作种类
- 视产品判断再决定是否扩展为训练计划 / 教练端能力

---

## 8. 当前结论
当前技术工作不应脱离产品与交互单独推进。接下来应先以 V1 PRD 和交互设计为准，对 demo 和后续研发内容进行重排和对齐。
